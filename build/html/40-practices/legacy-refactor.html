第 14 章　实战三：遗留系统改造与渐进重构 ​

每个程序员都梦想着从零开始写代码（Greenfield），但现实是，我们 80% 的时间都在和遗留代码（Legacy Code）打交道。

遗留代码通常意味着：没有文档、测试覆盖率低、变量命名像天书、原作者早已离职。

幸运的是，Claude Code 是对付遗留代码的神器。它不嫌弃代码乱，而且能瞬间读完几千行代码。本章将展示如何利用 Claude 进行安全的渐进式重构。

14.1 遗留系统盘点 ​

接手一个老项目的第一步，是搞清楚“它到底在干什么”。

14.1.1 利用 Ask Mode 绘制地图 ​

不要直接开始读代码。先用 Ask Mode 进行全局扫描。

User: “扫描 src/legacy/billing 目录。解释一下计费逻辑的主要流程。绘制一个 Mermaid 时序图。”

Claude 会阅读相关文件，总结出业务流程：

  1. 用户下单 -> processOrder()
  2. 计算折扣 -> calc_disc_v2() (注意这个 v2，暗示有历史包袱)
  3. 扣减库存 -> update_stock_final()

14.1.2 识别“坏味道” (Code Smells) ​

让 Claude 帮你找出最需要重构的地方。

User: “分析 OrderManager.java。指出其中复杂度最高、最难维护的方法，并说明原因。”

Claude 可能会指出：

  • calculateTotal 方法有 500 行，包含 20 个 if-else。
  • 存在大量的魔法数字（Magic Numbers）。
  • 数据库操作和业务逻辑混杂在一起。

14.2 设计渐进式重构方案 ​

重构最怕的是“步子迈太大，扯着蛋”。我们需要制定一个安全的计划。

14.2.1 建立安全网：补充测试 ​

黄金法则：在修改遗留代码之前，必须先有测试。

如果现在的代码没有测试，先让 Claude 写。

User: “我要重构 calculateTotal 方法。请先为它编写一组单元测试，覆盖主要的分支逻辑。不要修改原代码，只写测试。”

Claude 会生成测试用例。运行这些测试，确保它们通过（也就是捕捉到了当前的行文，哪怕是错误的行文，也要先“固化”下来）。

14.2.2 绞杀植物模式 (Strangler Fig Pattern) ​

不要一次性重写整个类。我们可以用“绞杀植物模式”，逐步替换。

  1. 创建新接口：定义一个干净的 IPricingService。
  2. 实现新逻辑：让 Claude 根据旧代码的逻辑，写一个新的实现类，使用现代的写法（比如策略模式代替 if-else）。
  3. 灰度切换：在旧代码中调用新接口。

14.3 执行与验证 ​

现在开始动手。

14.3.1 提取函数 (Extract Method) ​

这是最安全的重构手法。

User: “在 OrderManager.java 中，将 calculateTotal 方法里处理‘VIP 折扣’的 50 行代码提取为一个独立的方法 calculateVipDiscount。”

Claude 会自动处理变量的作用域，完成提取。

14.3.2 重命名变量 (Rename Variable) ​

把 a, b, tmp 变成有意义的名字。

User: “将 int d 重命名为 discountRate，将 float p 重命名为 finalPrice。”

14.3.3 转换技术栈 ​

有时你需要把代码从一种语言迁移到另一种（例如 Java -> Go，或 jQuery -> React）。

User: “读取 frontend/old-script.js。将其中的 DOM 操作逻辑重写为 React 组件。保持 CSS 类名不变。”

Claude 能够理解逻辑并进行“翻译”，而不仅仅是语法转换。

小结 ​

重构遗留系统就像拆弹。Claude Code 是你的拆弹专家：

  1. 它能看懂：快速理解复杂逻辑。
  2. 它能防爆：帮你编写测试用例作为安全网。
  3. 它手很稳：自动化执行重构操作，避免人为的拼写错误。

通过这三步——理解、加固、重构，你可以把任何一座“屎山”逐步改造成现代化的代码堡垒。

至此，我们的技术实战部分全部结束。最后一章，我们将聊聊人的问题——如何在团队中推广这些工具。
