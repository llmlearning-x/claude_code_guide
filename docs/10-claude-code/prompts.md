# 第 4 章　工程级 Prompt 设计：结构化与复用

![工程级 Prompt 设计：结构化与复用](../images/cc-prompts-cover.png)

提到 AI 编程，很多人第一反应是：“不就是把代码贴进去，然后问它怎么改吗？”
对于写个 Hello World 或者做道算法题，这当然没问题。但在复杂的工程环境中，随意的提问往往换来随意的回答——代码风格不统一、引入多余依赖、甚至破坏原有架构。

本章将教你如何写出**工程级 Prompt**。这种 Prompt 不再是随口一问的聊天，而是像写代码一样，有结构、有约束、可复用。

## 4.1 为什么你需要工程级 Prompt？

在工程实战中，我们对 AI 的输出有三个核心要求：
1.  **准确性 (Precision)**：不能幻觉，不能瞎编 API。
2.  **一致性 (Consistency)**：今天生成的代码和昨天生成的，风格应该一样。
3.  **依从性 (Compliance)**：必须遵守项目的 Lint 规则、命名规范和目录结构。

随意的 Prompt（“帮我写个登录页面”）很难同时满足这三点。而工程级 Prompt 通过显式的上下文和约束，强迫 AI “戴着镣铐跳舞”，从而输出高质量的代码。

---

## 4.2 Prompt 的解剖学：四大要素

一个优秀的工程级 Prompt，通常包含以下四个部分（CRIC 模型）：

### 4.2.1 Context (上下文)
告诉 Claude 它在什么环境里工作。
*   **角色设定**：你是一个资深前端架构师 / Python 后端专家。
*   **技术栈**：项目使用 React 18 + TypeScript + Tailwind CSS。
*   **业务背景**：这是一个电商后台管理系统。

### 4.2.2 Request (任务目标)
清晰、原子化的指令。
*   ❌ “优化一下这个函数。”（太模糊）
*   ✅ “重构 `calculateOrder` 函数，提取税费计算逻辑为独立函数，并添加 JSDoc 注释。”

### 4.2.3 Input Data (输入数据)
明确指出要处理哪些信息。
*   **相关文件**：`src/utils/tax.ts` 和 `src/types/order.ts`。
*   **参考代码**：参考 `src/utils/currency.ts` 的写法。
*   **错误日志**：粘贴具体的报错堆栈。

### 4.2.4 Constraints (约束条件)
这是最关键的部分，决定了输出是否“工程化”。
*   **风格约束**：使用函数式编程风格，禁止使用 `class`。
*   **依赖约束**：只能使用现有的 `lodash`，不能引入新库。
*   **输出格式**：只输出修改后的代码块，不要解释性文字。

## 4.3 常用 Prompt 模板模式

为了提高效率，我们可以总结出几种通用的 Prompt 模式。

### 4.3.1 “修复者”模式 (The Fixer)
用于修复 Bug 或错误。

> **模板**：
> “作为 [角色]，请分析以下 [错误日志/现象]。
> 这里的 [相关代码] 似乎有问题。
> 请找出根本原因，并提供修复方案。
> **约束**：保持原有函数签名不变；优先处理边界情况。”

### 4.3.2 “生成者”模式 (The Generator)
用于从零编写新功能。

> **模板**：
> “请根据 [需求描述/接口定义]，实现 [功能模块]。
> **技术栈**：[框架版本]。
> **约束**：
> 1. 组件必须拆分为 Presentational 和 Container。
> 2. 所有样式使用 Tailwind 类名。
> 3. 包含基本的单元测试。”

### 4.3.3 “审查者”模式 (The Reviewer)
用于代码审查和优化。

> **模板**：
> “请审查以下代码 [粘贴代码]。
> 重点关注：
> 1. 安全性（如 SQL 注入、XSS）。
> 2. 性能瓶颈。
> 3. 代码可读性。
> 请以列表形式输出 Review 意见，按严重程度排序。”

## 4.4 上下文管理技巧

在使用 Claude Code 时，如何高效地“喂”给它上下文是一门学问。

### 4.4.1 善用 `@file` 或 `/read`
不要手动复制粘贴几十个文件的内容。
*   使用 `claude "..." --file src/main.ts src/utils.ts`
*   或者在交互模式下直接引用文件名，Claude Code 通常能自动读取。

### 4.4.2 建立项目级 `.clauderc` (概念性)
虽然 Claude Code 目前主要通过 Prompt 传递上下文，但你可以维护一个项目级的 `PROMPT_GUIDE.md` 文件，里面写满项目的核心规范。
每次开始复杂任务前，先让 Claude 读取这个文件：
```bash
$ claude ask "Read PROMPT_GUIDE.md first, then help me refactor..."
```
这相当于给 Claude 加载了一个“项目特定的系统提示词”。

### 4.4.3 渐进式上下文
不要一次性把整个仓库丢给它。
1.  先让它 `ls` 或 `tree` 看目录结构。
2.  再让它读取核心配置文件（`package.json`, `go.mod`）了解依赖。
3.  最后只读取与当前任务直接相关的 3-5 个源文件。

## 4.5 从 Prompt 到 Skill

当你发现某个 Prompt 写得特别好，而且经常要用（比如“生成符合团队规范的 Commit Message”），你就应该把它升级为 **Skill**。

我们将在**下一部分（第 II 部分：Skill）**详细讲解如何把这些结构化的 Prompt 固化下来，变成团队可共享的工具。

> **思考题**：
> 回顾你最近一次让 AI 写代码的经历，检查一下你的 Prompt 是否包含了 CRIC 四要素？如果加上缺失的要素，结果会有什么不同？
